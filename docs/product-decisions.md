# 数探｜产品迭代与关键决策

本文记录数探从“能回答数据问题”到“能交付可核验分析结果”的产品取舍，用于说明产品定义、功能边界与评测设计之间的关系。

## 1. 从文件起步，而非先覆盖所有数据源

**问题**：运营和业务人员的临时分析往往从一份 Excel 或 CSV 开始；数据库、在线表格和 API 接入虽然价值更高，但还需要连接配置、权限与治理准备。

**选择**：MVP 以 CSV/Excel 的上传、预览、提问、执行、核验与追问为核心闭环；SQL 数据库、Google Sheets、HTTP API 和本地工作区作为后续数据接入能力。

**暂缓方案**：在首版同时完成全部企业数据源接入和协作治理。

**取舍**：文件入口降低了第一次使用和验证闭环的成本，但会带来文件版本、多 Sheet 与多文件关联的歧义。因此，产品将数据预览、活动数据范围与关联键确认放在分析前置流程中。

**验证信号**：用户能否在不写 SQL 的情况下完成“文件 → 分析 → 核验 → 追问”；文件解析成功率、首个有效结果耗时、数据预览后的提问完成率。

## 2. 让工具产生数据结果，而不是让模型直接给数字

**问题**：通用大模型可以快速生成看似合理的解释，但用户难以确认数字、排名和图表是否来自当前文件。

**选择**：大模型负责理解问题、拆解任务和解释结果；SQL、清洗、统计和图表工具负责实际执行。确定性结论必须能回溯到当前数据范围和工具结果。

**暂缓方案**：仅依靠 Prompt 约束模型在对话中直接生成分析结论。

**取舍**：工具调用会增加任务链路和等待时间，但可以把“回答得像分析”变成“结果可复核”。当工具失败、返回空结果或数据不支持结论时，系统展示失败原因、已完成部分和下一步，而不以确定性口吻补全答案。

**迭代依据**：离线评测使用真实工具调用验证工具路由、SQL 执行与结果正确性、图表适配和安全拒答。Prompt B 增加证据优先、字段检查、避免重复失败调用和因果边界约束后，在独立测试集中 SQL 结果正确率提升 6.25 个百分点，端到端任务完成率提升 5.00 个百分点。

## 3. 自动执行默认开启，只确认会改变结果的歧义

**问题**：每一步都要求用户确认会把简单取数变成繁琐流程；但静默猜测时间范围、指标定义或关联字段，又可能导致结果错误。

**选择**：字段、时间范围、数据来源和指标口径充分时直接执行；仅对会显著改变结果的时间范围、统计口径、关联键和多个候选数据源发起确认。

**暂缓方案**：每次工具调用都由用户批准，或完全不询问并由模型默认补全。

**取舍**：产品以最少的追问换取可用性，同时将关键确认作为结果可信度的一部分。确认后的范围和口径会在同一会话的后续追问中继承，用户仍可主动更正。

**验证信号**：澄清请求率、澄清后的任务完成率、用户改写口径的比例、因口径或关联字段导致的结果错误率。

## 4. 展示执行摘要，不展示完整模型推理

**问题**：用户需要核对数据来源、计算方式和失败原因，但不需要阅读冗长、难稳定复核的内部推理。

**选择**：在对话中展示数据范围、工具名称与状态、SQL 或计算摘要、结构化结果、图表来源和错误信息；不展示模型内部思维链。

**暂缓方案**：只给最终结论，或完整暴露模型的内部推理过程。

**取舍**：执行摘要兼顾可读性与可追溯性。对大结果，用户可以打开表格、图表或导出产物继续核对，而不让对话承载全部明细。

**验证信号**：结果依据查看率、结果采纳率、用户对数据范围和计算口径的理解程度、结果纠错率。

## 5. 用个人知识库沉淀口径，不把聊天记录当作知识库

**问题**：有效客户、复购率、渠道归因等定义常在后续分析中重复出现；只依赖会话上下文容易产生口径漂移。

**选择**：个人知识库保存指标定义、业务规则、背景知识和可检索文档，并在分析中按用户私有范围检索；历史会话保存分析过程和结果，两者分别服务于口径复用与任务回看。

**暂缓方案**：仅保存聊天记录，或在缺少权限模型时直接建设团队共享知识库。

**取舍**：个人知识库先解决单用户的复用问题；团队共享、权限协作与知识治理进入后续版本。知识内容需要支持编辑、启停和确认，避免过期规则影响分析。

**验证信号**：知识库检索命中率、口径复用率、用户修订知识条目的频率、因历史口径冲突触发确认的比例。

## 6. 产品工作与后续迭代

产品工作围绕用户场景、MVP 边界、交互规则、验收标准和离线评测展开：先用文件分析闭环限定首版范围，再用工具执行与结果依据建立可信交付；评测将字段理解、工具路由、SQL 结果、图表选择、拒答和重复调用等失败类型拆开定位，并将修正方向落实到 Prompt、工具契约、Schema 校验和交互确认。

下一步优先验证字段理解与 SQL 结果校验，再根据文件场景的完成情况决定多文件关联和更多数据源的投入顺序。
